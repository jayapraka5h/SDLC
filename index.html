Question 2 ‚Äî Phases Deep Dive
Q "Can you explain all 6 phases of SDLC with an example?"

Say this first: "Sure! Let me explain each phase using the example of building a food delivery app."

1
Requirements Phase
"In this phase, a Business Analyst meets the client and collects all the requirements ‚Äî what features the software should have. The output is a requirements document like FRS or BDD. Example: The BA asks the QuickBite founder ‚Äî do you want OTP login? GPS tracking? Cash on delivery?"
2
Analysis Phase
"The System Analyst checks if the requirements are technically and financially feasible. They select the technology and do resource and time planning. The output is the SRS document. Example: They decide to use React Native, 10 developers, 6 months timeline."
3
Design Phase
"The Chief Architect creates a high-level design ‚Äî breaking the project into modules. The Technical Lead does low-level design ‚Äî breaking modules into sub-modules and writing pseudo code. Output is the Technical Design Document. Example: The app is split into Customer App, Restaurant App, and Delivery App."
4
Coding Phase
"Developers write the actual code based on the design document, following coding standards like proper indentation, naming conventions, and commenting. Output is the source code. Example: One team builds login, another builds payment, another builds GPS tracking."
5
Testing Phase
"Test engineers verify the software against requirements. They write test cases, execute them, log defects, and re-test after fixes. This cycle continues until the product is defect-free. Output is a quality product. Example: Testers check ‚Äî what happens if payment fails midway? Does GPS update live?"
6
Delivery & Maintenance Phase
"The deployment engineer installs the software in the client's environment. After delivery, any issues are fixed by the maintenance team. Example: The app is published on Play Store. Later, a bug is found and fixed in the next update."

Pro Tip After explaining all 6 phases, end with: "This structured approach ensures the software is built correctly, on time, and within budget." ‚Äî This closing line makes you sound professional.

Common Follow-Up Questions
‚ùì What is a prototype in SDLC?
"A prototype is a rough model of the software shown to the client to confirm requirements. It is used when requirements are unclear, to gain client confidence."
‚ùì What is the difference between SRS and FRS?
FRS ‚Äî written by BA after meeting client, covers business requirements. SRS ‚Äî written by System Analyst, covers technical feasibility and system specs.
‚ùì What happens if requirements are unclear?
"The Engagement Manager is responsible. He either deals extra cost with the client or builds a prototype to gather clear requirements."
‚ùì Who does testing in SDLC?
"Test Engineers or QA Engineers. They read the requirements document, write test cases, execute them on builds, and log defects until the product is defect-free."
‚ùì What is high-level vs low-level design?
High-level ‚Äî done by Chief Architect. Divides project into modules. Low-level ‚Äî done by Technical Lead. Divides modules into sub-modules + pseudo code.
‚ùì What is maintenance in SDLC?
"After delivery, if any issue occurs in the live software, the maintenance team fixes it. Some clients have a dedicated maintenance team working permanently at their site."
